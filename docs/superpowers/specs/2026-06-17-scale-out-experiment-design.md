# Scale-out experiment: distributed clients vs DS & ursula on Kubernetes

**Date:** 2026-06-17
**Status:** Approved design, pending implementation plan
**Repo:** `ds-rust-bench`
**Track:** 2 of 2 — scale-out experiment. Depends on
[Track 1: single-node comparison](2026-06-17-single-node-bench-design.md), which
builds `ds-bench` v0 and settles the durable-streams backend mapping.

## Purpose

Run a multi-node, scale-out benchmark of Durable Streams servers under synthetic
load generated by a **fleet of distributed clients on Kubernetes**, for
**competitive positioning**. The experiment exercises the DS protocol's hot paths —
client fan-out (SSE) and write throughput foremost — and measures how throughput
and tail latency behave as both server nodes and client load scale out.

## Systems under test

| System | Topology | Notes |
| --- | --- | --- |
| durable-streams (Rust) | single node now; **multi-node when it lands** (Phase 3) | the system being positioned |
| durable-streams (Node/TS) | single node | our own baseline + protocol-portability proof; expected far slower (research: ~20–250× behind Rust) |
| ursula | **1, 3, 5 nodes** (multi-Raft) | competitor's scale-out story is its whole pitch; deployed via its existing Helm chart |

S2 Lite is explicitly out of scope.

## Locked decisions

| Decision | Choice |
| --- | --- |
| Goal | Competitive positioning (strict fairness + reproducibility) |
| IaC target | **Kubernetes, cloud-agnostic** (portable manifests + Helm; concrete cluster chosen at run time) |
| Workload client | **`ds-bench`** (built in Track 1), extended here; containerized and fanned out across client pods |
| First-run workloads | All four: write-throughput, SSE fan-out, mixed, catch-up replay |
| Durability alignment | Matched durable-to-disk + object-store offload (same as Track 1) |
| Object store | MinIO in-cluster by default (portable); swappable for cloud S3 |
| Observability | Prometheus + Grafana (kube-prometheus-stack) live; authoritative percentiles from merged HDR |

## Why Kubernetes + our own client (resolved tension)

Generic advice says "k8s pushes you to k6+xk6-sse," which is a fidelity regression
for SSE end-to-end latency. That only applies if k6 *is* the client. Because we run
**our own Rust `ds-bench`** containerized as the client, we keep full SSE
fidelity (payload-embedded ns timestamp → `now − sent`, native HDR) **and** get
k8s-native orchestration. The highest per-node throughput ceiling (Rust, no GC)
also keeps the load generator from becoming the bottleneck when measuring server
tail latency.

## Architecture / components

### ds-bench extensions (built here, on top of Track 1 v0)
- **Serialized-HDR output** — each client pod writes its HDR histogram (the
  `hdrhistogram` crate's serialized form) to a shared sink; merge is associative
  and lossless, giving authoritative cross-fleet percentiles.
- **New workloads:**
  - **catch-up-read** — our own **protocol-faithful** replay workload: N clients
    simultaneously catch-up read a stream from offset 0 until `Stream-Up-To-Date`,
    hot (tail) vs cold (object store). This **replaces** ursula-bench's `bootstrap`,
    which is built on ursula's `/bootstrap` + `/snapshot` endpoints that are **not in
    the Durable Streams protocol** (see Track 1). Run symmetrically against all
    backends via plain catch-up reads.
  - **mixed** — concurrent writers + catch-up readers + live SSE subscribers on
    shared streams (recreates ursula's unpublished `mixed` phase). Most realistic.
  - (catalog extras — stream-cardinality, resume/resilience — deferred unless the
    first run motivates them.)
- **Backend mapping** for durable-streams (Rust + Node) and ursula, reusing the
  Track-1 api-style work.

### Distribution layer (k8s)
- `ds-bench` containerized; fanned out as a **Job array / small controller** across
  client nodes, scaled to keep the server-under-test saturated (load generator must
  not bottleneck).
- **Aggregation:** pods write serialized HDR to a shared sink (object store / PVC);
  a **coordinator** merges → unified percentiles + ops/s. Methodology borrowed from
  OpenMessaging Benchmark's HdrHistogram-merge approach (we reuse the *method*, not
  its Java workers).
- **Live telemetry:** Prometheus remote-write + Grafana dashboards during runs;
  merged HDR is the source of truth for published numbers.

### Server deployments (k8s)
- **durable-streams (Rust / Node):** our own manifests (Deployment/StatefulSet +
  Service), `--tier s3` / cold tier to the in-cluster object store.
- **ursula:** its **existing Helm chart** (`charts/ursula`) at 1/3/5 replicas,
  single-voter→multi-voter Raft, cold tier to the same object store.
- **Object store:** MinIO in-cluster (default) or cloud S3.

## Experiment matrix (first run)

- **Workloads:** write-throughput, SSE fan-out, mixed, catch-up replay.
- **Systems × topology:** DS-rust ×{1}, DS-node ×{1}, ursula ×{1, 3, 5}.
  DS-rust ×{3, 5} joins in Phase 3 when multi-node lands.
- **Sweeps:**
  - writes: payload 100 B / 1 KB / 16 KB; client-pod count scaled to saturation.
  - fan-out: subscribers 100 → 1k → 10k across client pods.
  - mixed: fixed writer/reader/subscriber ratio, scaled together.
  - replay: backfill size sweep, hot (tail) vs cold (object store) reads.
- **Headline outputs:**
  - throughput + p99/p999 vs **server-node count** (scale-out efficiency curve).
  - fan-out latency vs **subscriber count**.
  - single-node head-to-head (DS-rust vs ursula-1) carried from Track 1.

## Fairness controls (carried from Track 1)

- Matched durable-to-disk: ursula `[raft.wal] backend = "disk"` (single-voter at
  N=1; real quorum at N=3/5); durable-streams fsync. Both offload to the same
  object store.
- Pinned, identical pod/node resources — a fixed node-pool instance type
  (c7g-class or equivalent) and explicit CPU/memory requests+limits; one
  server-under-test per measured run.
- Identical `ds-bench` parameters across systems; **group-commit-symmetry**
  disclosure in the writeup (both servers coalesce fsyncs).
- **Honesty note (critical):** at 3/5 nodes ursula pays cross-node Raft replication
  that single-node DS-rust does not. So multi-node throughput is **not**
  apples-to-apples until DS-rust has multi-node. Report two **separate** stories
  until Phase 3: (a) single-node head-to-head, (b) ursula's own scale-out curve.
  Do not headline "DS beats ursula" using a single-node DS number against a
  3-node ursula number.

## Open items (resolved in planning/implementation)

1. **durable-streams backend mapping** in `ds-bench` (the `--api-style` fit) —
   shared with Track 1, settled there first.
2. **Coordinator + HDR-merge mechanism** — shared-sink format (object store vs PVC),
   k8s Job-array vs a small custom controller, and how the coordinator discovers and
   waits for all pods.
3. **Concrete k8s distribution + node-pool instance type** for runs (kept out of the
   portable manifests; chosen per run).
4. **MinIO in-cluster vs cloud S3** for the offload tier per run.
5. **Client-fleet sizing** — how many pods/nodes are needed to saturate each server
   topology without the generator becoming the bottleneck (determined empirically;
   `ds-bench`'s own CPU headroom is monitored during runs).

## Phasing

- **Phase 2 (this spec, first run):** single-node DS-rust + DS-node vs ursula 1/3/5,
  all four workloads, on k8s with a distributed client fleet.
- **Phase 3:** DS-rust multi-node (when it lands) → true 3-node ↔ 3-node and
  5-node ↔ 5-node comparisons, including ursula's replication in the head-to-head.

## Success criteria (Phase 2)

- IaC stands up, on a k8s cluster: the object store, a chosen server topology, and a
  scalable `ds-bench` client fleet.
- All four workloads run distributed; per-pod HDR histograms merge into unified,
  authoritative percentiles; Grafana shows live run telemetry.
- A rendered report shows, per workload: throughput + p99/p999 vs server-node count,
  and fan-out latency vs subscriber count, across DS-rust, DS-node, and ursula
  1/3/5 — with the single-node vs multi-node honesty split made explicit.
